## 本地常驻 MCP RAG 系统详细设计说明书 (V6.0 - 生产级全特性规范)
本说明书严格基于 GitHub 开源生态中高星、生产级 RAG MCP 项目（如 shinpr/mcp-local-rag、0xrdan/mcp-rag-server）的真实演进规范编写。系统取消低效的 Stdio 桥接器模式，全面采用原生的 HTTP/SSE (Server-Sent Events) 作为长连接传输层，并完整实现了 Model Context Protocol (MCP) 官方规范中的五大核心特征：Tools（工具）、Resources（资源）、Prompts（提示词模板）、Sampling（反向采样）和 Roots（根目录感知）。
通过这五大特性的有机编排，系统构建了一个前后台双向联动、具备隐私隔离和自主可观测性的闭环 RAG 生态系统。
------------------------------
## 1. 技术堆栈选型

* 通信协议层：Model Context Protocol (MCP) SDK v1.2+ (Python)，JSON-RPC 2.0 规范，SSE 异步网络传输。
* 网络框架层：FastAPI v0.111+, Uvicorn v0.29+ (ASGI 异步网关服务器)。
* 核心存储层：LanceDB v0.7+ (基于 Rust 构建的嵌入式、列式无服务器向量引擎)，PyArrow v16.0+。
* 特征提取层：Ollama API 接口，默认绑定 bge-m3 多语言混合检索嵌入模型（1024 维度）。
* 多模态解析层：PyPDF v4.2+ / Python-Docx v1.2+ / Pandas v2.2+。
* 日志审计层：Loguru v0.7+ (支持多线程异步写入、按天轮转及压缩的结构化日志库)。

## 2.1 MCP 五大特性功能规格

* F01-Tools 原子工具箱（Tools Framework）：向连接的 AI 宿主（Client）暴露符合开源规范的 7 个原子化动作函数调用接口，允许 AI 智能体自主执行模糊检索、链式邻近块读取以及按需写入。
* F02-Resources 动态数据看板（Resources Registry）：开放全局 URI 资产路径，以标准 JSON-RPC 2.0 格式向客户端提供只读的知识边界盘点与实时状态诊断，防止 AI 盲目检索。
* F03-Prompts 动态提示词注入（Prompts Engine）：提供预设的快捷指令模板（如代码库全局评审、长文档多模态对比），动态注入到编辑器的快捷操作菜单中，规范大模型的 RAG 检索流程（SOP）。
* F04-Sampling 反向采样协作（Sampling Callback）：打破传统单向调用，允许本地 RAG 守护进程在遇到复杂、高噪的本地文件时，反向请求前台大模型进行文本清洗与关键词提取，直接“白嫖”前台高参数模型的推理能力。
* F05-Roots 动态隐私隔离监控（Roots Perception）：在握手初始化时动态获取 AI 编辑器当前打开的项目工作区路径，自动将 File Watcher 的监控焦点和 LanceDB 的命名空间（Namespace）切换至该路径，实现多项目间的数据强隔离。

## 2.2 高级多模态 ETL 解析流水线

* F06-基于 Pandas 的 Excel 矩阵序列化器（Excel ETL）：将半结构化表格行无损转化为具有显式列字段声明的元数据对（如 "列名A: 值A; 列名B: 值B;"），确保表格结构边界完好。
* F07-物理段落流式 PDF 解析器（PDF ETL）：基于双换行符进行物理段落逻辑重构，自动执行正则表达式过滤页眉、页脚与页码。
* F08-对象树 Word 结构化提取器（Word ETL）：基于 python-docx 遍历段落与表格对象树，保证段落层次结构清晰，并将文档内嵌表格同步转化为矩阵序列化文本。
* F09-动态句尾标点防截断分块算法（Smart Chunking）：硬性约束 Chunk Size = 600 字符，Overlap Size = 60 字符，并在切分边界处智能检索最近的完整句尾标点进行语义切分。

## 2.3 高性能存储与混合检索引擎

* F10-LanceDB 无服务器高性能持久化（Serverless Storage）：依托 Rust 列式文件存储格式和内存映射（mmap）机制实现数据落盘。闲置时物理内存占用控制在极低水平（≤ 35MB），读写无锁并发。
* F11-IVF-PQ 与 BM25 双索引构建（Dual Indexing）：针对文档表同时自动化构建面向高泛化语义的 IVF-PQ 密集向量索引，和面向特定型号/编码/数字的 BM25 稀疏倒排索引(FTS)。
* F12-基于 RRF 的混合检索重排器（Hybrid RRF Ranker）：并行执行双索引召回，引入环境变量参数 RAG_HYBRID_WEIGHT 控制检索权重，并利用倒数排名融合（RRF）无监督算法合并双侧得分。

## 2.4 可观测性与自动化运维

* F13-内联级异步文件变更防抖监控（Watcher & Debounce）：实时捕获监控目录内文件的创建、修改、删除。内置 1.0s 防抖状态机，待文件写入句柄完全释放并保持静止时再触发特征计算，对于删除文件执行物理擦除。
* F14-全链路 Trace 耗时审计日志（Observability Logging）：基于 loguru，精确追踪从文件变更、ETL 解析、Embedding 获取到最终混合检索各阶段的毫秒级 Latency，生成完整的审计链。
* F15-日志自动化滚动与深度清理（Log Management）：每日凌晨 00:00 触发日志文件物理切分，并全自动执行 zip 深度压缩与超过 10 天的陈旧日志物理抹除。

## 4. MCP 特性矩阵详述与核心数据契约
本章定义了本系统常驻后台核心暴露给大模型客户端的五大核心特性契约标准。
## 🛠️ 4.1 Tools（开源原子化工具清单）
系统向连接的 AI 宿主暴露符合开源规范的 7 个原子化动作函数调用接口（协议映射键：tools/call）。
## 4.1.1 数据写入类工具 (Data Ingestion Tools)

* ingest_file：系统级别的物理写操作。接收用户指定的物理文件绝对路径（必须属于 Roots 约束的路径），触发多模态解析器，完成切片、向量化并打入 LanceDB。
* 参数 Schema：{"type": "object", "properties": {"path": {"type": "string", "description": "本地操作系统中待建立索引的文件的绝对路径"}}, "required": ["path"]}
* ingest_data：内存级的无文件写操作。允许 AI 将对话中捕获的任意 Markdown 文本、剪贴板数据或网络爬虫抓取的 HTML 网页字符串直接注入向量库，用于构建长短期记忆备忘录。
* 参数 Schema：{"type": "object", "properties": {"content": {"type": "string", "description": "待注入的纯文本或 Markdown 字符串"}, "metadata": {"type": "object", "properties": {"source": {"type": "string", "description": "该数据的来源标识，如网页 URL 或 Clipboard"}}}}, "required": ["content"]}

## 4.1.2 上下文检索类工具 (Retrieval Tools)

* query_documents：系统的核心读操作。执行底层的密集向量与稀疏全文混合检索，结合 RRF 算法获取前 K 个高相关性分块，每个召回分块严格透传 file_id 和 chunk_id。
* 参数 Schema：{"type": "object", "properties": {"query": {"type": "string", "description": "提炼过的核心自然语言问题或关键词短语"}, "limit": {"type": "integer", "description": "返回的高相关性上下文切片最大数量上限", "default": 4}, "scope": {"type": "string", "description": "可选字段，指定进行过滤的文件名关键词"}}, "required": ["query"]}
* read_chunk_neighbors：跨分块语义桥接器。当 AI 通过 query_documents 召回的文本片段信息不全（如代码行在交界处被切断、逻辑缺少前因后果）时，AI 主动调用此工具向上或向下捞取物理相邻的分块，彻底防止长文本中断章取义。
* 参数 Schema：{"type": "object", "properties": {"file_id": {"type": "string", "description": "目标文件的唯一标识路径"}, "chunk_id": {"type": "integer", "description": "当前已知的基准分块序号"}, "direction": {"type": "string", "enum": ["prev", "next", "both"], "description": "读取邻近块的方向"}}, "required": ["file_id", "chunk_id", "direction"]}

## 4.1.3 资产管理与生命周期工具 (Lifecycle Tools)

* list_files：自省资产盘点动作。列出当前本地向量库中已成功构建索引的所有文件列表、切片分布总量及最后修改时间戳。参数 Schema 为空对象。
* delete_file：物理索引擦除动作。从 LanceDB 物理表中单向移除指定文件的所有 Chunk 指纹，并连带重构 FTS 全文倒排索引。
* 参数 Schema：{"type": "object", "properties": {"file_id": {"type": "string", "description": "要执行物理擦除的文件路径"}}, "required": ["file_id"]}
* status：系统级运行健康诊断。返回当前 LanceDB 物理表的行数、物理磁盘空间占用、日志健康度以及特征层网络连接指纹。参数 Schema 为空对象。

## 📂 4.2 Resources（只读数据源）
系统向大模型暴露静态资产结构和诊断看板（协议方法：resources/list，resources/read），指导 AI 在执行昂贵的工具检索前先明确“知识边界”。

* localrag://system/status
* 功能描述：动态向 AI 暴露当前常驻服务的底层健康快照。
   * MimeType：application/json
* localrag://files/list
* 功能描述：将当前向量数据库中已经索引完毕的所有文件绝对路径清单作为静态虚拟文件暴露给大模型。
   * AI 应用场景：当用户提问“你知道我的 XXX 项目设计书放在哪吗？”，大模型在静默状态下直接读取此 Resource。若在清单中直接看到该路径，则调用 query_documents 进行深度检索；若未看到，则直接利用 Tool 提示用户进行建索，大幅缩短无效 RAG 检索开销。

## 📝 4.3 Prompts（动态提示词模板）
系统通过协议方法（prompts/list，prompts/get）向 AI 编辑器前台注入 RAG 专属的快捷操作指令，规范大模型的知识召回工作流（SOP）。

* review-codebase (代码库全局评审模板)
* 动态注入参数：{"focus_area": "性能优化 / 漏洞排查 / 逻辑重构"}
   * 底层下发的 Prompt 载荷：
   
   你现在是资深的本地代码库架构师。你当前所处的项目空间已通过 Roots 挂载到本地常驻 RAG 服务器上。
   针对用户要求审查的领域 {{focus_area}}，你必须严格遵循以下 SOP 操作：
   1. 必须优先调用 `query_documents` 工具模糊搜索项目中包含关键逻辑的模块。
   2. 如果发现召回的代码片段被切断，你必须连续调用 `read_chunk_neighbors` 补全上下文。
   3. 在你的最终回答中，必须每一处代码片段都严格注明 `file_id` 和 `chunk_id` 以供用户追溯。

## 🔄 4.4 Sampling（反向采样协作机制）
在高级多模态 ETL 解析阶段，系统通过协议方法（sampling/createMessage）实现常驻后台反向差遣前台大模型的倒置控制链。

* 技术痛点：本地常驻服务为了保持轻量（物理内存 ≤ 35MB），不适合在本地常驻运行复杂的 LLM 模型。然而，当用户丢入包含大量格式混乱、文本高噪的扫描件 PDF 或凌乱的原始日志时，纯代码规则（正则表达式）的分块提取效果极差。
* 开源解决逻辑（Sampling Workflow）：
1. 本地监控发现新文件 dirty_report.pdf。
   2. PDF ETL 流式抽取底层纯文本，发现文本存在严重的信息错位和乱码高噪。
   3. 后台服务挂起当前文件的写入事务，反向向当前连接的 AI 宿主编辑器（如 Claude Code）发起 sampling/createMessage 请求：
   
   {
     "messages": [{
       "role": "user",
       "content": "请作为高效的数据清洗专家，帮我将以下这段高噪乱码的原始文本进行格式规整。剔除页眉页脚和无关乱码，保留核心段落，并为该段落提取出 5 个最核心的实体关键词，以标准的 Markdown 格式返回。文本内容如下：\n[高噪原始文本...]"
     }],
     "maxTokens": 1024
   }
   
   4. 前台大模型（如 Claude 3.7 Sonnet）在云端或前台完成高性能推理清洗，将近乎完美的结构化文本回传给常驻后台。
   5. 后台服务无缝接收响应，利用 Ollama 生成向量特征并打入 LanceDB。
* 核心价值：完全“白嫖”前台大模型的能力反哺本地知识库，让本地 RAG 具备低消耗、高精度的多模态数据清洗能力。

## 🗂️ 4.5 Roots（根目录隐私隔离感知）
系统通过订阅协议方法（notifications/initialized 和 roots/list），动态界定数据隐私边界。

* 动态握手行为：当用户在 Cursor 中打开项目工作区 A（路径：/Users/admin/projects/A）时，AI 编辑器作为 MCP 客户端，在初始化连接建立后，会将该工作区路径作为 Roots 列表通知本地常驻后台。
* 物理隔离策略：
1. 文件监控动态寻址：后台服务的 FileSystemEventHandler 收到通知后，立刻销毁旧的监听器，将 watchdog 监控焦点实时绑定到项目 A 的路径下，开启增量同步。
   2. 数据库命名空间强隔离：系统利用 LanceDB 的虚拟表名称或 Schema Metadata，自动为当前项目开辟专有的存储命名空间（Namespace = project_A）。
   3. 当大模型调用 query_documents 工具时，系统底层强行追加作用域过滤条件：where("root_space = '/Users/admin/projects/A'")。
* 核心价值：当用户从项目 A 切换到项目 B 时，系统全自动切换数据上下文，确保项目 A 的核心资产绝不会在项目 B 的对话中被意外召回和泄露，彻底锁死隐私边界。

## 5. 核心混合检索（Hybrid Search）与重排算法
在工具 query_documents（工具三）的底层核心中，系统采用无监督的 RRF (Reciprocal Rank Fusion) 算法，并结合环境变量进行精细化权重控流。对于出现在密集检索分支（ANN 拓扑搜索）和稀疏检索分支（BM25 FTS）中的文档 d，其综合得分的推导公式规范为：
$$RRF\_Score(d) = \frac{w}{k + R_{\text{dense}}(d)} + \frac{1 - w}{k + R_{\text{sparse}}(d)}$$ 

* 常量常数 k：系统采用开源最佳实践值 60，平滑低排名对整体权重的扰动。
* 权重调节因子 w：由环境配置项 RAG_HYBRID_WEIGHT 动态注入（默认 0.7）。
* 当 w = 0.7 时，系统处于企业最优配比，兼顾深层语义并向精确的货号、代码关键字符号倾斜。
